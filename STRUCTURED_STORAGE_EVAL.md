# P3-03 IndexedDB / SQLite 结构化数据迁移评估

> 本文是迁移评估，不代表已经实施结构化迁移。当前版本仍保持 `localStorage` 主数据 + IndexedDB 图片外置的架构。

## 1. 当前存储架构

当前《月之暗面》仍是本地离线单页应用，所有脚本通过普通 `<script src>` 顺序加载，不依赖 npm、CDN、构建步骤或远程 API。

### 1.1 主数据

- 主业务数据保存在 `localStorage` 的 `KEY = "osddDidLocalJournal.v2"`。
- 旧版本数据会从 `OLD_KEY = "osddDidLocalJournal.v1"` 读取并迁移到当前结构。
- 偏好设置保存在 `PREF_KEY = "osddDidLocalJournal.prefs.v1"`。
- 账本记录独立保存在 `LEDGER_KEY = "moonLedger.records.v1"`。
- `load()` 会整包读取 JSON、解析、执行 `migrate()`，再进入内存中的完整 `data` 对象。
- `save()` 会把完整 `data` 对象整包 `JSON.stringify()` 后写回 `localStorage`。

### 1.2 图片

- 图片已外置到 IndexedDB：数据库 `moon-images`，版本 `1`，object store `images`，`keyPath = "id"`。
- 消息图片通常使用 `msgimg-${message.id}`。
- 成员头像通常使用 `avatar-${member.id}`。
- 房间背景图通常使用 `roombg-${room.id}`。
- 运行时主数据只保存 `imageId` / `avatarId` / `backgroundId` 等引用，不把大图 DataURL 长期塞回主 JSON。

### 1.3 完整 JSON 备份

- 完整 JSON 导出会先深拷贝主数据和账本记录。
- `hydrateImagesForJsonExport()` 只在导出副本中从 IndexedDB 读取 Blob，并转成 `imageData` / `avatarData` / `backgroundData`。
- 运行中的 `data` 不会因为导出而重新内联图片。
- JSON 导入会先解析、迁移、校验，再由 `externalizeImagesAfterJsonImport()` 把内联 DataURL 写入 IndexedDB，设置对应图片 ID，并删除导入对象里的 DataURL。
- 导入失败不会覆盖当前 `data`。

### 1.4 消息完整性

- `messageIntegrity(m)` 使用消息的 `seq`、`id`、`roomId`、`speakerId`、`speakerName`、`kind`、`text`、图片字段和 `createdAt` 等字段计算校验。
- 当消息有 `imageId` 时，完整性计算使用 `imageId` 和 `_imgVer = 2`，不再依赖内联 `imageData`。
- `nextSeq` 会根据现有 `messages` 中的最大 `seq` 重算。
- 清空消息后，下一条新消息的序号会回到 `0001`。

## 2. 当前性能结论

### 2.1 P3-01 基线

P3-01 在 C 档真实压力数据下记录：

- 主数据 `localStorage` 约 `3.10 MiB`。
- 单个常用房间 `3,000` 条消息时，`render()` 平均约 `168 ms`。
- 同场景聊天切换 / `renderChat()` 约 `155 ms`。
- `.msg` DOM 约 `3,000` 个。
- 时间线聚合约 `9,400` 条来源，生成约 `59 ms`。

结论：P3-01 阶段第一瓶颈是聊天区一次性渲染大量 DOM。

### 2.2 P3-02 后状态

P3-02 引入消息虚拟滚动后，同样单房间 `3,000` 条消息验收结果：

- `render()` 平均约 `4.23 ms`。
- `renderChat()` 平均约 `2.40 ms`。
- 实际 `.msg` DOM 约 `60` 个。
- 搜索跳转第 `10` / `1500` / `2500` 条消息均能定位并高亮。
- 高级搜索、时间线、复盘报告、JSON 导入导出、图片外置语义均通过回归。

因此，当前第一瓶颈已经不再是聊天 DOM 渲染。

### 2.3 后续主要风险

未来更可能出现的性能和稳定性风险是：

- `localStorage` 主 JSON 体积继续增长，接近浏览器配额或写入上限。
- 每次 `save()` 都写整包 JSON，数据越大，写入越重。
- 启动 / 导入时仍需要整包解析和 `migrate()`。
- 高级搜索、时间线、月度回顾、复盘报告仍依赖完整内存对象聚合。
- 图片已外置到 IndexedDB，但结构化文本数据仍集中在主 JSON。

## 3. 建议迁移触发条件

建议使用以下阈值判断是否进入结构化迁移实施阶段：

| 条件 | 建议动作 |
| --- | --- |
| 主数据 `localStorage` 超过 `5 MiB` | 进入观察，评估体积提示、压缩、拆分保存 |
| 主数据接近 `8-10 MiB` | 优先评估结构化迁移，避免配额和写入风险 |
| 消息超过 `20,000` 条且 `save()` / `load()` 明显变慢 | 评估 `messages` object store |
| 高级搜索 / 时间线 / 复盘报告超过 `200 ms` | 评估索引、分表或范围查询 |
| 需要 PWA、桌面壳、移动壳或多设备迁移 | 重新评估持久化层边界 |
| 用户需要真实全文搜索 | 评估 IndexedDB 索引或 SQLite FTS |

这些阈值不是硬性故障线，而是进入设计和原型阶段的信号。

## 4. 方案 A：继续 localStorage 主数据

### 优点

- 当前实现简单，调试直观。
- 完整 JSON 备份语义清晰，用户容易理解。
- `data` 始终是完整内存对象，现有渲染、搜索、时间线、复盘报告逻辑成本低。
- 没有 IndexedDB 事务边界、分页加载、异步一致性等额外复杂度。
- 更符合当前本地离线单页应用的轻量定位。

### 缺点

- 所有数据一次性 `load()`、`migrate()`、`save()`。
- 主 JSON 变大后，写入耗时和失败风险都会增加。
- 浏览器 `localStorage` 配额和实现差异不可完全控制。
- 任何小改动都需要重写整包数据。
- P3-02 只减少 DOM 渲染压力，不减少主数据体积。

### 适合阶段

当前阶段仍适合继续使用此方案：

- P3-01 C 档主数据约 `3.10 MiB`，尚未接近建议迁移线。
- P3-02 已解决当前最明显的聊天渲染卡顿。
- 当前功能高度依赖完整内存对象，立即迁移会带来较大回归面。

### 风险

- 用户长期积累大量消息、任务、照护记录、前台日志后，主 JSON 会持续增长。
- 频繁保存可能在低性能设备上造成短暂停顿。
- 大备份导入导出仍需要整包处理。

### 可先做的优化

- 统计 `save()` 调用频率和写入耗时。
- 对连续输入或批量操作做保存节流 / 批量保存。
- 在设置或导出面板提示主数据体积。
- 评估可选压缩，但需确保备份可读性和失败恢复策略。
- 对大备份导入导出做 spot check，持续记录耗时和失败率。

## 5. 方案 B：IndexedDB 结构化主数据

可评估把以下结构拆成 IndexedDB object stores：

- `rooms`
- `members`
- `messages`
- `polls`
- `handoffNotes`
- `frontingLogs`
- `tasks`
- `careLogs`
- `careChecklist`
- `systemProfile`
- `systemProfileVisibility`
- `externalSystemCards`
- `memberRelations`

### 优点

- 避免主结构化数据继续受 `localStorage` 单包体积限制。
- `messages` 可以分页加载，聊天虚拟滚动可进一步只取需要范围。
- 时间线可以按时间范围查询，减少全量内存扫描。
- 搜索可以先按 store 或索引缩小范围。
- 未来可以更自然地支持增量保存、批量事务和恢复检查。

### 缺点

- 迁移复杂度高。
- 当前大量功能假设 `data` 是完整内存对象。
- `render()`、搜索、时间线、月度回顾、复盘报告、JSON 导入导出都会被影响。
- 旧 JSON 导入导出需要适配新持久化层，但仍要保持完整备份语义。
- IndexedDB API 异步化会把很多同步路径变成异步路径，回归范围很大。
- 若设计不当，可能出现部分 store 写入成功、部分失败的一致性问题。

### 必须设计清楚的内容

- IndexedDB schema version 和升级路径。
- 每类数据的 key、索引、排序字段和反查字段。
- transaction 策略，尤其是导入、批量删除、批量迁移。
- 迁移失败回滚策略。
- JSON 导入失败时不能覆盖当前数据。
- 完整 JSON 备份仍必须能生成，且不能丢字段。
- 图片 hydrate / externalize 仍必须只改变导出副本或导入对象。
- 离线可用性不能降低。
- `messageIntegrity` 和 `nextSeq` 的语义必须保持稳定。

### 推荐分阶段路线

1. 先只拆 `messages`。
   - 消息数量增长最快，也是最容易超过体积和渲染阈值的部分。
   - 可先保留 `rooms` / `members` / 设置类数据在主 JSON 中。
   - 需要兼容搜索跳转、聊天虚拟滚动、消息校验和完整备份。

2. 再拆 `frontingLogs` / `tasks` / `careLogs`。
   - 这些数据会进入时间线、复盘报告和接续面板。
   - 拆分后需要设计时间范围查询和多源聚合接口。

3. 最后再拆 `members` / `rooms` / `systemProfile` 等核心结构。
   - 这些对象被大量 UI 和业务逻辑引用。
   - 过早拆分会放大回归风险。

### 当前判断

不建议现在直接全量迁移到 IndexedDB 结构化主数据。当前更合理的是先做监控、节流、体积提示和技术原型，等真实数据越过阈值后再实施。

## 6. 方案 C：SQLite

SQLite 更适合以下场景：

- 进入原生桌面或移动壳。
- 需要真正的全文搜索 FTS。
- 需要复杂结构化查询、排序、筛选和索引。
- 数据量远大于当前浏览器单页应用预期。
- 产品形态不再只是静态浏览器页面。

### 限制

- 当前静态浏览器页面没有原生 SQLite。
- 引入 SQLite 通常意味着 WASM SQLite 或 Electron / Tauri / Flutter / 原生壳。
- WASM SQLite 会引入依赖、加载体积、持久化适配和备份复杂度。
- 原生壳会改变当前无构建、无依赖、直接打开或本地 HTTP 运行的简单性。

### 当前判断

SQLite 不应作为当前 Web 版本的优先方案。它更适合作为未来桌面壳、移动壳或重度全文搜索版本的研究方向。

## 7. 推荐路线

### 当前阶段建议

1. 暂不实施结构化迁移。
2. 保持 `localStorage` 主数据 + IndexedDB 图片外置。
3. 继续观察真实主数据体积。
4. 优先做 `save()` 写入频率统计。
5. 优先做主数据体积提示和风险警告。
6. 可选评估数据压缩，但不要影响完整 JSON 可恢复性。
7. 持续做大备份导入导出 spot check。
8. 如果真实数据超过阈值，再启动 P3-03 实施版。

### 未来实施优先级

| 阶段 | 内容 | 目标 |
| --- | --- | --- |
| P3-03a | 存储体积监控 UI / 警告 | 让用户知道主数据何时接近风险区 |
| P3-03b | `save()` 节流和批量保存 | 减少整包写入频率 |
| P3-03c | `messages` 拆到 IndexedDB object store 的技术原型 | 验证分页加载、完整备份和校验兼容 |
| P3-03d | 时间线 / 搜索基于 IndexedDB 范围查询 | 降低全量内存聚合压力 |
| P3-03e | SQLite / 原生壳研究 | 面向桌面壳、移动壳和 FTS 场景 |

## 8. 迁移风险清单

| 风险 | 说明 |
| --- | --- |
| `messageIntegrity` 不能随意变 | 任何字段变化都可能让历史消息显示校验异常 |
| `nextSeq` 不能错 | 清空消息、导入备份、删除消息后都要保持序号连续语义 |
| JSON 完整备份不能丢字段 | 备份必须继续包含当前所有业务数据和账本数据 |
| JSON 导入失败不能覆盖当前数据 | 迁移到多 store 后更要保证原子性或可回滚 |
| 图片 hydrate / externalize 不能破坏 | 导出应 hydrate 副本，导入应外置图片并清理 DataURL |
| 隐私桶和脱敏导出不能误公开 | 复盘报告和脱敏导出不能因新查询层泄露私密内容 |
| 接续面板 / 时间线 / 复盘报告依赖多源聚合 | 拆 store 后要重新设计统一聚合接口 |
| 高级搜索依赖全量内存数据 | 结构化迁移后要决定搜索走内存、索引还是分页扫描 |
| 任务 / 交接 / 投票之间有关联 | 拆分保存不能破坏关联 ID 和显示语义 |
| 历史前台日志和删除成员 ID 可能仍需显示 | 查询层不能只依赖当前成员列表存在 |

## 9. 建议下一步选择

- 如果想继续稳定化：做 P3-03a 存储体积监控。
- 如果想继续性能：做 P3-03b `save()` 节流和批量保存。
- 如果想增强安全：做可选加密备份。
- 如果想发布：做 PWA / 桌面壳评估。

## 10. 结论

当前不建议马上把结构化主数据迁移到 IndexedDB 或 SQLite。

P3-02 已经把当前最明显的聊天渲染瓶颈降下来；P3-01 C 档主数据约 `3.10 MiB`，还没有到必须迁移的程度。下一步更适合先建立体积监控、保存频率统计和大备份 spot check，让真实数据增长趋势决定是否进入结构化迁移实施。

推荐路线是：短期继续 `localStorage` 主数据 + IndexedDB 图片；中期优先优化 `save()` 和体积提醒；长期在消息量、主 JSON 体积或搜索/时间线耗时越过阈值后，先做 `messages` object store 原型，再评估更完整的 IndexedDB 结构化迁移。SQLite 暂时只作为未来原生壳或全文搜索方向。
