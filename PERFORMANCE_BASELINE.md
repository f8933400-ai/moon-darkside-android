## 5. 时间线与聊天

- 时间线全量聚合：A 200 条 / 33.9 ms；B 2,050 条 / 41.7 ms；C 9,400 条 / 58.9 ms。
- 时间线显示截断：B、C 均只显示最近 300 条，截断正常。
- 日期筛选：A 19.8 ms；B 33.7 ms；C 43.2 ms。
- 成员筛选：A 21.3 ms；B 17.9 ms；C 39.6 ms。C 命中 314 条，显示 300 条。
- 聊天切换：C 档热门房间 3,000 条消息渲染 154.6 ms；普通房间 / 私聊约 16-17 ms。
- `renderChat()` 的 `_renderChatSeq` 在快速切换 race 测试中保持最终房间正确，没有观察到旧异步渲染覆盖新房间。

## 6. 导出 / 导入 / 图片

- 复盘报告 `review-md`、`review-txt`、脱敏 + 隐藏成员名 + 排除私聊均生成成功。
- 复盘报告生成时 `imageStore.getImageBlob` 读取次数为 0，未读取图片 Blob，也未导出图片 DataURL。
- JSON 完整备份会按预期 hydrate 图片：每档 3 个测试图片引用，JSON 导出读取 3 次 Blob，并包含 DataURL。
- JSON 导入后 3 个测试图片引用均能从 IndexedDB 读回。
- 本轮只使用 3 张 1x1 PNG 测试图片，未覆盖大量真实图片压力。

## 10. 观察到的问题

### 阻塞

- 无。

### 高

- 无。

### 中

- C 档热门房间 3,000 条消息时，聊天区单次渲染约 155 ms，`render()` 平均约 168 ms，已经足以产生可感知卡顿。
- C 档 localStorage 主数据约 3.10 MiB；在常见浏览器配额下尚可运行，但继续增长会接近风险区。

### 低

- 启动“可操作”耗时主要由固定 3 秒免责声明等待决定，纯数据加载不是启动瓶颈。
- 搜索 100 条、时间线 300 条截断均正常，但宽泛查询会隐藏更早结果。
- C 档脱敏复盘报告约 225 ms，明显高于普通复盘报告，原因主要是成员名替换和排除私聊处理。

## 11. 是否建议进入虚拟滚动

建议优先做消息虚拟滚动。

理由：时间线在 9,400 条聚合来源下仍约 59 ms，且已有 300 条显示截断；消息区在单房间 3,000 条时已经达到约 155 ms，并拖高全局 `render()`。时间线虚拟滚动可以暂缓，IndexedDB 结构化迁移建议单独评估。
